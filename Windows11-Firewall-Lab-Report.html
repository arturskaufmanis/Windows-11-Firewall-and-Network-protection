📋 README — Project Overview

This project documents a complete hands-on lab exercise covering Windows 11 firewall configuration and network protection. The lab was designed to build practical experience with the exact scenarios encountered by SOC analysts and system administrators when investigating reports of blocked or inaccessible websites.

Objectives

  • Understand the difference between Windows Defender Firewall outbound rules and Windows Defender Network Protection (SmartScreen)
  • Create and manage firewall rules using both PowerShell and the wf.msc GUI
  • Observe and document the browser error messages produced by each type of block
  • Read and interpret firewall drop logs (pfirewall.log) and Event Viewer audit trails
  • Remediate firewall blocks and restore connectivity
  • Enable, test, and disable Network Protection (SmartScreen) in both Audit and Enforce modes

All commands were run in Windows PowerShell (Administrator), presented in execution order with expected output noted.

PHASE 1 Baseline & DNS Resolution

Step 1.1 — Confirm target site loads

Navigate to https://example.com in Edge. Confirm page loads before making any changes.

Step 1.2 — Resolve DNS for target

PowerShellResolve-DnsName example.com
⚠
NoteExpected output: 2× AAAA records (IPv6) and 2× A records (IPv4). Note all 4 IPs — you must block all of them or the browser will use an unblocked one.

Figure 1 — Resolve-DnsName showing 4 addresses for example.com

PHASE 2 Block via PowerShell

Step 2.1 — Create outbound block rule

PowerShellNew-NetFirewallRule -DisplayName "LAB-Block-ExampleCom" `
  -Direction     Outbound `
  -Action        Block `
  -RemoteAddress 104.18.26.120,104.18.27.120,2606:4700::6812:1a78,2606:4700::6812:1b78 `
  -Profile       Any `
  -Enabled       True
ℹ
InfoNo output confirmation prompt. Rule is immediately active. All 4 IPs (both IPv4 and IPv6) must be included in a single rule.

Figure 2 — New-NetFirewallRule executed successfully, PrimaryStatus: OK

Step 2.2 — Test connectivity (confirm block is active)

PowerShellTest-NetConnection -ComputerName 104.18.26.120 -Port 443
✅
Expected ResultTcpTestSucceeded : False — confirms TCP connection is being dropped by the firewall.

Figure 3 — Browser showing ERR_NETWORK_ACCESS_DENIED after rule creation

🔬 Analysis & Key Findings

Error Signature Comparison

Understanding which error a user reports is the first triage step. The table below maps errors to their causes:

Browser ErrorCauseBehaviour
ERR_NETWORK_ACCESS_DENIEDFirewall outbound rule blockImmediate — WFP intercepts at socket level before packet leaves machine
ERR_CONNECTION_TIMED_OUTFirewall block (silent drop)Slow ~30s — packet sent but silently discarded
Red SmartScreen pageNetwork Protection / SmartScreenInstant — reputation check at DNS/URL layer
ERR_NAME_NOT_RESOLVEDDNS block or NXDOMAINInstant — domain not resolved at all
ERR_CONNECTION_REFUSEDActive port rejectionInstant — RST packet received

SOC Analyst Triage Workflow

When a user reports "website won't open", follow this diagnostic chain:

  1. DNS check: Resolve-DnsName <domain> — does it resolve to an IP?
  2. TCP test: Test-NetConnection -ComputerName <IP> -Port 443 — does TcpTestSucceeded return True or False?
  3. If False: Check pfirewall.log for DROP entries to that IP on port 443
  4. If DROP entries found: Get-NetFirewallRule | Where-Object { $_.Action -eq 'Block' } to find the blocking rule
  5. Identify rule source (who created it, when) via Event ID 2082 in the Firewall event log
  6. Remove rule if appropriate: Remove-NetFirewallRule -DisplayName '<name>'
  7. If browser shows red SmartScreen page instead: check Network Protection state and SmartScreen settings in Windows Security

Observations from Lab

  • example.com returned 4 addresses (2 IPv4 + 2 IPv6). Blocking only the IPv4 addresses would have allowed the browser to connect via IPv6 — all addresses must be blocked simultaneously.
  • The ERR_NETWORK_ACCESS_DENIED error (vs the expected ERR_CONNECTION_TIMED_OUT) occurred because Edge attempted IPv6 first, which was blocked at the WFP layer before the TCP handshake could begin — producing an immediate rejection rather than a timeout.
  • Event ID 2082 recorded both GUI (mmc.exe) and command-line (netsh.exe) changes with precise timestamps — demonstrating the value of the firewall audit log for forensic reconstruction.
  • Network Protection (value=0) was disabled by default on this standalone Windows 11 machine — consistent with Microsoft's default configuration for non-enterprise deployments.
  • The GUI SmartScreen toggle (App & browser control → Reputation-based protection) controls the Edge-level block. The master Network Protection setting requires PowerShell or Group Policy.
